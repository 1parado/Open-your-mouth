# AI 口语老师产品设计方案

## 1. 产品定义

AI 口语老师不是通用聊天应用，也不是传统 LMS。它的核心形态是一个“可配置多老师角色的实时口语陪练产品”。

用户进入产品后，先选择老师，再进入具体练习。每个老师都不是一段 prompt，而是一个完整的数字教学角色，包含：

- 形象
- 声音
- 人设
- 教学风格
- 难度范围
- 擅长场景
- 纠错策略

产品的长期目标不是只做语音问答，而是形成“老师驱动”的学习体验：用户能长期跟随某位老师练习，并感知到稳定的人格、视觉形象和教学反馈。

## 2. 产品目标

### 核心目标

1. 给用户一个低门槛、可持续使用的英语口语练习入口。
2. 把 AI 能力包装成“老师”而不是“模型”。
3. 为后续数字人形象输出预留统一的老师配置体系。
4. 在保持系统简洁的前提下，支持多老师、多风格、多练习场景扩展。

### 非目标

- 不优先做复杂课程平台。
- 不优先做教师排课、班级管理、作业系统。
- 不优先做大而全内容商城。

## 3. 目标用户

### 用户类型

- 自主学习者：希望随时开口练英语。
- 场景训练用户：重点练面试、出国、旅游、商务等口语场景。
- 纠音用户：重点关心发音、流利度、表达准确性。

### 用户核心需求

- 快速开始，不想先学复杂功能。
- 想要“像老师一样”的反馈，而不是冷冰冰的模型回复。
- 希望老师有明确风格，比如温柔、严格、商务、美音纠音。
- 希望练习后获得简洁但明确的改进建议。

## 4. 产品形态

产品分为三个层级：

### 4.1 当前可交付形态

- Web 端
- 移动端
- 语音对话为主，文本为辅

### 4.2 进阶形态

- 2D 数字人老师
- 老师说话时展示口型、表情、动作
- 用户可切换老师形象与声音

### 4.3 完整形态

- 多老师体系
- 多场景练习
- 练习记录与成长反馈
- 轻量课程引导
- 订阅或按角色解锁

## 7. 用户端功能设计

## 7.1 首页

目标是 10 秒内让用户开始练习。

首页应突出：

- 继续上次练习
- 选择老师
- 热门练习场景
- 最近反馈摘要

不建议首页堆复杂课程导航。

## 7.2 老师广场

用户按“老师”而不是“模型”选入口。

展示维度：

- 老师头像 / 数字人封面
- 一句话风格标签
- 擅长领域
- 口音类型
- 难度等级
- 是否支持数字人模式

筛选条件：

- 初级 / 中级 / 高级
- 日常 / 旅游 / 面试 / 商务
- 美音 / 英音
- 温柔 / 严格 / 高互动

## 7.3 老师详情页

这是转化页，负责让用户决定“跟谁练”。

建议模块：

- 老师介绍
- 试听声音
- 试看形象
- 教学风格
- 擅长场景
- 用户适配等级
- 推荐练习入口

## 7.4 练习准备页

进入练习前，用户只需要完成最少配置：

- 选择老师
- 选择场景
- 选择模式：语音模式 / 数字人模式
- 可选选择难度

默认自动带出老师推荐场景，不强迫用户做复杂设置。

## 7.5 实时练习页

这是产品最重要的页面。

页面结构建议：

- 中央：老师形象或头像
- 底部：麦克风、结束、重试、字幕开关
- 侧边或底部抽屉：对话文本、关键词提示、网络状态

核心交互规则：

1. 用户点击开始或自动进入会话。
2. 老师主动开场，降低冷启动成本。
3. 用户讲话后，系统先转写，再生成老师回复。
4. 老师通过语音回复；数字人模式同步口型和表情。
5. 当用户打断时，系统停止老师播报并切换到监听态。

## 7.6 练习结果页

结果页要简洁，不要像考试系统。

建议输出：

- 总体评价
- 发音评分
- 流利度评分
- 重点纠错 3 条
- 更自然表达建议 2-3 条
- 本次新词 / 句型
- 下一次推荐练习

## 7.7 历史记录页

用户可查看：

- 练习时间
- 练习老师
- 练习场景
- 总体评分变化
- 历次反馈摘要

后续可增加“继续和该老师练习”快捷入口。

## 8. 教学设计

## 8.1 老师风格体系

每位老师必须可配置以下教学维度：

- 语气：温柔 / 严厉 / 专业 / 活泼
- 主动性：高引导 / 中引导 / 少引导
- 纠错时机：实时纠错 / 回合后纠错 / 结束后总结
- 语言比例：全英文 / 中英结合 / 以中文解释为辅
- 难度控制：保守 / 适中 / 激进

## 8.2 反馈层级

系统反馈不要一刀切，建议分层：

1. 会话中轻提示
2. 单轮结束后纠错
3. 整场结束后总结

这样能兼顾练习流畅性和教学质量。

## 8.3 场景体系

建议首批场景：

- 自我介绍
- 日常聊天
- 旅游出行
- 餐厅点餐
- 酒店入住
- 机场值机
- 工作面试
- 商务会议

每个场景绑定：

- 目标词汇
- 关键句型
- 用户任务目标
- 老师引导脚本

## 9. 数字人设计

## 9.1 数字人不是独立产品，而是老师的一种呈现层

因此数字人能力应挂在 `teacher` 之下，而不是单独做一套完全平行的产品结构。

## 9.2 数字人模式分级

### Level 1

- 静态头像 + 实时语音

### Level 2

- 2D 半身老师 + 嘴型同步 + 基础表情

### Level 3

- 2D/3D 数字人 + 情绪驱动 + 动作模板 + 镜头调度

## 9.3 数字人运行链路

1. LLM 生成老师回复文本
2. TTS 生成老师语音
3. 数字人引擎读取音频与情绪标签
4. 输出口型、表情、动作
5. 客户端播放音视频结果

建议保留降级路径：数字人渲染失败时自动回退为纯语音模式。

## 9.4 老师形象配置维度

- 封面图
- 立绘 / 半身 / 全身
- 说话动画模板
- 待机动画模板
- 情绪映射规则
- 字幕样式
- 品牌主题色

## 10. 管理后台设计

## 10.1 老师管理

后台可以创建和编辑老师：

- 基本资料
- 形象
- 声音
- 系统提示词
- 教学风格
- 启用状态

## 10.2 场景管理

后台可按老师绑定场景，允许：

- 复用通用场景
- 覆盖默认 prompt
- 配置目标词汇与评分标准

## 10.3 对话质检

运营或产品可查看：

- 会话回放
- 转写内容
- 老师回复
- 用户中断次数
- 错误评测样本

这部分对后续调优非常关键。

## 10.4 供应商管理

后台或配置中心应能管理：

- LLM Provider
- ASR Provider
- TTS Provider
- Pronunciation Provider
- Digital Human Render Provider

产品层看到的是老师；系统层看到的是老师背后的能力组合。

## 11. 推荐的完整版本功能边界

完整版本建议包含 6 个产品域：

1. 用户与鉴权
2. 老师系统
3. 场景练习系统
4. 实时会话系统
5. 反馈与成长系统
6. 内容与老师运营后台

其中“课程系统”可以作为次级能力存在，但不应喧宾夺主。

## 12. 指标设计

### 北极星指标

- 每周完成口语练习的活跃用户数

### 核心业务指标

- 首次练习启动率
- 单次练习完成率
- 次日 / 7 日复练率
- 单用户每周练习次数
- 老师收藏率
- 不同老师的转化率
- 数字人模式使用率

### 质量指标

- 语音识别成功率
- 首响应时延
- TTS 生成时延
- 用户中断率
- 评分结果可解释性满意度

## 13. 完整版本阶段规划

### Phase 1：老师驱动的语音陪练

- 老师广场
- 老师详情
- 实时语音对话
- 基础发音评测
- 历史记录

### Phase 2：数字人老师

- 2D 数字人展示
- 嘴型同步
- 情绪驱动
- 多老师形象切换

### Phase 3：成长与内容体系

- 场景分级
- 个性化推荐
- 周报 / 阶段报告
- 老师带练路线

### Phase 4：商业化与运营

- 订阅体系
- 角色解锁
- 高级老师付费
- 企业 / 机构版本

## 14. 对当前项目的落地建议

基于现有代码结构，建议做以下调整：

1. 把“老师”提升为一等核心实体，不再以课程为产品入口。
2. `frontend/web` 首页改成“选老师 -> 选场景 -> 开始练习”。
3. `api` 优先建设 `teachers`, `teacher_profiles`, `scenarios`, `practice_sessions`。
4. `realtime-orchestrator` 先支持语音模式，再为数字人模式预留事件与媒体输出协议。
5. `provider-gateway` 继续保留，因为它非常适合承接 TTS、发音评测和未来数字人渲染供应商。
6. 结果页只保留关键反馈，不做厚重报告中心。

## 15. 一句话结论

这个产品的完整定义应当是：

**一个以“可配置老师角色”为核心、支持实时语音互动，并可逐步升级为多数字人老师体系的 AI 口语陪练产品。**
