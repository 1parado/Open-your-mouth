# 7 天推进计划

## 使用方式

这个计划不是旧版“全微服务一次性铺开”的开发安排，而是围绕当前产品方向做的短周期推进：

**先把老师驱动的主流程做稳，再逐步把后端化和实时化补上。**

## Day 1：老师产品入口

### 目标

- 登录后首页变成老师卡片页
- 至少 3 位老师
- 不同老师有不同文案、voice、prompt

### 交付物

- 老师配置文件
- 老师卡片 UI
- 老师详情 / 会话入口

## Day 2：老师会话页

### 目标

- 老师可以主动开场
- 用户可以文字输入
- 老师可以返回差异化回复
- 老师回复可以自动 TTS 播放

### 交付物

- 会话页 UI
- LLM + TTS 接入
- 基础消息记录

## Day 3：语音输入

### 目标

- 用户可以通过浏览器语音识别直接说话
- 识别失败时可回退为文字输入
- 基础状态提示完整

### 交付物

- 语音输入按钮
- Listening 状态反馈
- 语音失败兜底

## Day 4：老师系统后端化

### 目标

- 老师不再只写在前端
- 首页老师数据由 API 提供
- 场景和会话入口开始后端化

### 交付物

- `teachers`、`scenarios`、`practice_sessions`
- 基础查询接口

## Day 5：练习反馈

### 目标

- 会话结束后显示轻量反馈
- 反馈包括评分、重点问题、下一步建议

### 交付物

- 会话结果页
- 历史记录页升级
- 基础反馈存储

## Day 6：实时链路预备

### 目标

- 评估并拆出 Realtime Orchestrator
- 设计 WebSocket 事件协议
- 明确中断控制逻辑

### 交付物

- 实时链路设计稿
- 事件定义
- 技术拆分计划

## Day 7：下一阶段准备

### 目标

- 明确数字人接入边界
- 明确后台老师管理边界
- 明确 Phase 2 的接口与数据模型

### 交付物

- 数字人字段设计
- 后台需求清单
- 下一周迭代计划

## 每日检查项

- 首页是否还在围绕“老师”组织
- 新能力是否服务主会话流程
- 是否避免重新回到“工具页集合”
- 是否给未来数字人保留了数据字段或接口边界

## 一句话结论

这一周的重点不是把所有基础设施铺满，而是把“选老师 -> 开口练 -> 听老师回复”做成产品。
